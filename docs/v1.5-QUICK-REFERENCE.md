// NEW: Render paragraphs as <p> elements (not pre-wrap text)
function renderParagraphs(text) {
  return text.split(/\n\n+/).map(para => {
    const div = document.createElement('div');
    div.textContent = para.trim();
    return `<p>${div.innerHTML}</p>`;
  }).join('');
}

## Common Questions

**Q: Why reduce max_tokens to 1200?**
A: Plain Finnish translations rarely exceed 800 tokens. 1200 provides 50% headroom. Saves 70% on API costs.

**Q: Why remove Google Fonts?**
A: Google Fonts requests send user data (IP, browser, referrer) to Google. System fonts look nearly identical and are already on user's device.

**Q: Why paragraph rendering?**
A: Semantic HTML (`<p>` elements) is better for accessibility and CSS styling than `white-space: pre-wrap`.

**Q: Do I need to change my code?**
A: No. The changes are backend and frontend optimizations. If you're using the API, no changes needed.

**Q: Will this break my extension?**
A: No. The extension automatically uses new paragraph rendering. System fonts render similarly.

**Q: What about the VPS changes?**
A: Required for proper rate limiting. Without them, rate limiting won't work correctly (all requests appear from same IP).

## Troubleshooting

**Problem:** Startup assertions fail
```
AssertionError: OPENROUTER_API_KEY is required
```
**Solution:** Set environment variables before starting:
```bash
export OPENROUTER_API_KEY=sk-...
export ALLOWED_ORIGIN=http://localhost:3000
```

**Problem:** Rate limiting doesn't work
```
All requests seem to bypass 30/hour limit
```
**Solution:** Configure Nginx and systemd as documented above.

**Problem:** Extension text doesn't render with proper spacing
```
All text runs together (no paragraph breaks)
```
**Solution:** Verify backend is returning `\n\n` between paragraphs in response.

**Problem:** Fonts look different than before
```
Serif font appears different on macOS/Windows
```
**Solution:** System fonts render platform-specifically. This is expected and acceptable.
