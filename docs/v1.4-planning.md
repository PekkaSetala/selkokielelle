# selkokielelle.fi — v1.4 Planning Document

**Prepared:** 2026-03-06
**Baseline:** v1.3.0 (live at selkokielelle.fi)
**Audience:** Internal — for management review

---

## Executive Summary

v1.3.0 delivered a stable, well-structured product: clean UI, accurate privacy disclosures, correct SEO signals, and a functional AI translation pipeline. The service is live and working.

v1.4 is not about adding features. It is about making the service resilient, observable, and defensible — the groundwork that needs to be in place before the user base grows. None of these items are glamorous. All of them are necessary.

The concerns are grouped into four categories: **Operations**, **Backend**, **Frontend UX**, and **Repository / Process**. Each item carries a priority rating:

- **P0** — Fix before any growth or promotion. A failure here causes real damage (downtime, cost overrun, data loss).
- **P1** — Fix in v1.4. Absence degrades reliability or user trust in a measurable way.
- **P2** — Worthwhile but deferrable. Low risk if not shipped in v1.4.

---

## 1. Operations

### 1.1 — No uptime monitoring `[P0]`

**Current state:** There is no alerting in place. If the service goes down — server crash, OOM kill, uvicorn process failure, nginx misconfiguration after a future change — nobody will know until a user reports it or someone manually checks.

**Risk:** Silent downtime. The service could be broken for hours or days without the team being aware.

**Recommendation:** Set up UptimeRobot (free tier) or Better Stack to ping `https://selkokielelle.fi` every 5 minutes. Configure email or SMS alerts on failure. This is a 15-minute task and costs nothing.

**Dependency:** Requires the `/health` endpoint from item 2.2 to be a more meaningful signal than a static HTML response.

---

### 1.2 — Log rotation not confirmed `[P1]`

**Current state:** The backend runs under systemd, which routes logs to `journald`. By default, `journald` does apply some limits, but the `SystemMaxUse` cap is not confirmed to be set on this VPS. On a small server (likely 20–40 GB disk), uncontrolled log growth from a high-traffic period or a bug causing excessive error output could fill the disk and bring the service down.

**Risk:** Disk-fill causing service outage. Silent degradation.

**Recommendation:** SSH into the server, confirm `/etc/systemd/journald.conf` has `SystemMaxUse=200M` or similar. One config change, one restart of `systemd-journald`. Document the confirmed value.

---

### 1.3 — No dependency version pinning `[P1]`

**Current state:** It is not confirmed that `backend/requirements.txt` pins exact package versions (e.g. `fastapi==0.115.0` vs `fastapi>=0.100`). If the server environment is ever rebuilt — new VPS, disaster recovery, onboarding a second developer — a `pip install -r requirements.txt` with unpinned versions could pull in a breaking change silently.

**Risk:** A future redeployment installs incompatible package versions and breaks the service. The failure may not be obvious at deploy time — it could surface as a subtle runtime error.

**Recommendation:** Run `pip freeze > requirements.txt` in the current production venv to capture exact pinned versions. Commit this. Redeployment becomes deterministic.

## 2. Backend

### 2.1 — No rate limiting `[P0]`

**Current state:** The `/api/translate` endpoint accepts unlimited requests from any IP. There is no throttling at the application level and no confirmed rate limiting at the nginx level.

**Risk:** Two distinct failure modes:

1. **Cost exposure.** Every request hits OpenRouter, which bills per token. A single malicious or misconfigured client could run up a significant bill in minutes. The current system has no circuit breaker.
2. **Service degradation.** A flood of requests (intentional or not) will exhaust the server's resources and deny service to legitimate users.

**Recommendation:** Add `slowapi` (a FastAPI-compatible rate limiting library, one dependency, ~10 lines of code) to enforce a per-IP limit of, for example, 10 requests per minute. This is not a security silver bullet, but it eliminates the most obvious abuse vector and protects the OpenRouter budget. Nginx-level limiting can complement this.

---

### 2.2 — No health endpoint `[P1]`

**Current state:** The only endpoint is `POST /api/translate`. There is no way for a monitoring tool, load balancer, or ops check to verify the service is alive without sending a real translation request (which costs money and produces log noise).

**Risk:** Uptime monitoring is either absent (see 1.1) or wasteful. No clean way to confirm the service is running without side effects.

**Recommendation:** Add `GET /api/health` returning `{"status": "ok"}` with HTTP 200. This is three lines of code. It gives monitoring tools a clean, free, side-effect-free signal and documents the service's operational interface.

---

### 2.3 — Error handling does not distinguish failure types `[P1]`

**Current state:** If OpenRouter returns a non-200 response for any reason — rate limit hit, authentication failure, model unavailable, billing issue — the user sees a generic Finnish error message: `"Jokin meni pieleen, yritä uudelleen"`. The backend logs nothing structured about the upstream error. There is no differentiation between:

- A transient error (retry would succeed)
- A persistent error (retrying will not help — e.g. auth failure, budget exceeded)

**Risk:** When a real problem occurs upstream, debugging it is slow because there is no structured signal. Users are given retry guidance even when retrying is futile.

**Recommendation:**
- Log the OpenRouter HTTP status code and response body (truncated) to journald on non-200.
- Return differentiated user messages: transient errors suggest retrying; persistent errors do not.
- This does not require exposing internal details to the user — it only requires the backend to act on the upstream status code before composing the response.

---

### 2.4 — Client-side timeout has no UI feedback `[P2]`

**Current state:** The backend timeout is 15 seconds (`httpx` client). The frontend has no corresponding timeout. If a request takes 10–15 seconds, the spinner runs indefinitely from the user's perspective. There is no message indicating the service is slow but working.

**Risk:** Users abandon the session or submit duplicate requests, assuming the tool is broken.

**Recommendation:** Add a client-side timeout (e.g. 20 seconds via `AbortController`) that replaces the spinner with a message such as "Tämä kestää odotettua kauemmin — odota hetki tai yritä uudelleen." Low effort, meaningful UX improvement on congested days.

## 3. Frontend UX

### 3.1 — tietosuoja.html missing SEO tags `[P1]`

**Current state:** The main `index.html` received a canonical tag, meta description, and updated title in v1.3.0. `tietosuoja.html` has none of these. If search engines index the privacy page — which they will — it will appear with no description and a generic title, inconsistent with the main page's SEO posture.

**Risk:** Weak search presence for branded queries. Minor trust signal missing.

**Recommendation:** Add to `tietosuoja.html`:
```html
<title>Tietosuoja – Selkokielelle</title>
<meta name="description" content="Tietosuoja ja tietoja selkokielelle.fi-palvelusta." />
<link rel="canonical" href="https://selkokielelle.fi/tietosuoja.html" />
```
Five minutes of work. Closes the SEO gap between the two pages.

---

### 3.2 — No user feedback signal on translation quality `[P2]`

**Current state:** Once a translation is returned, the user's only options are to copy it or ignore it. There is no mechanism for the user to signal whether the output was useful. The team has no data on translation quality in the wild.

**Risk:** The system prompt and model may be performing poorly for certain input types (long texts, highly technical content, informal language) with no signal back to the team.

**Recommendation:** Add a minimal thumbs up / thumbs down beneath the output. This does not need to persist to a database initially — even a client-side log or a lightweight serverside counter (no text stored, only a boolean + timestamp) would give directional signal. Consider this a prerequisite for any future prompt engineering work.

## 4. Repository and Process

### 4.1 — `docs/` is gitignored: release notes are local only `[P1]`

**Current state:** The `.gitignore` file excludes `docs/` from version control. This was originally intended to suppress internal build session logs (`phase-*-complete*.md`). However, `docs/v1.3.0.md` (the release notes just created) and this planning document are also excluded. They exist only on the local machine.

**Risk:** Release notes and planning documents are lost if the local machine is lost. New contributors or managers cannot access project history from the repository. There is no audit trail of decisions.

**Recommendation:** Revise `.gitignore` to exclude only build session artefacts, not release documentation:

```gitignore
# Build session artefacts — local only
docs/phase-*
docs/*-complete*
```

Track `docs/v*.md` and `docs/*-planning.md` in the repo. This change is one line in `.gitignore` and a single commit.

---

### 4.2 — No `CHANGELOG.md` `[P2]`

**Current state:** Version history exists only as individual `docs/vX.Y.Z.md` files and git commit messages. There is no single file that gives a reader — new developer, manager, external stakeholder — a linear view of what changed across the project's lifetime.

**Risk:** Onboarding friction increases as the project grows. No single source of truth for "what does this version do compared to the last."

**Recommendation:** Create `CHANGELOG.md` at the repo root, following the Keep a Changelog format. Backfill entries for v1.2 and v1.3.0 from existing docs and commit messages. Maintain it with each future release. Low ongoing cost, high long-term value.
